# Kickoff

## Что обсудили

Владелец описал задачу: бот для HH, который сам листает ленту, матчит
вакансии под два резюме, откликается с индивидуализированным
сопроводительным и отвечает рекрутерским ботам в чате до тех пор, пока
те не считают разговор закрытым. Мотивация прямая — рынок такой, что
ручной отклик уже не имеет смысла, надо набирать массу.

Сразу обозначены жёсткие ограничения:

- HH агрессивно банит за автозапросы → работаем на уровне браузера, а
  не API.
- Один аккаунт владельца.
- Проект open-source, поэтому всё операторо-специфичное (резюме,
  cookies, ключи) — за пределами репо.
- Postgres.
- Крутится только на локальном ноуте владельца, никаких VPS.

Обсудили два развилки:

1. **Claude Skills или свой код?** Решено: свой код. Skills — это
   расширения интерактивной сессии Claude Code, у них нет независимого
   runtime. Бот должен жить сам по себе и реагировать асинхронно. См.
   D-003.
2. **DOM или vision/computer use?** Решено: DOM через Playwright. HH
   стабильный, селекторы надёжны, vision был бы в 10–100× дороже без
   выигрыша по обходу защиты. См. D-001, D-002.

LLM (Claude API) оставляем только для трёх задач: матч резюме ↔
вакансия, генерация сопроводительного, ответы в чате.

## Что решили

- Зафиксированы D-001 … D-007 в [decisions.md](decisions.md).
- Базовая архитектура расписана в [architecture.md](architecture.md):
  два воркера (`feed` и `chat`), Postgres, persistent Chrome profile

- План разбит на 7 фаз в [roadmap.md](roadmap.md). Текущая — Phase 0
  (skeleton репо).
- Создан каталог `llm-docs/` как рабочая память: владелец имеет к нему
  доступ, ассистент читает на старте каждой сессии и пишет на ходу.
- Имя папки `llm-docs/` (kebab-case) вместо буквального "LLM Docs" —
  это open-source convention, пробелы в путях ломают cross-platform
  скрипты. Владелец может переименовать, если важно.
- Язык: проектные доки на английском (репо публичный), session-логи на
  русском. Зафиксировано в [conventions.md](conventions.md).

## Уточнение по ходу сессии — источник вакансий

Владелец уточнил: feed-воркеру не нужен поисковой URL с фильтрами. На
главной странице HH есть персонализированная лента, которую HH сам
подбирает под резюме залогиненного пользователя. Бот просто открывает
главную, листает эту ленту и заходит в карточки.

Что обновлено:

- Зафиксировано как [D-008](decisions.md#d-008--2026-05-26--main-page-personalized-feed-as-the-job-source).
- `architecture.md` — feed-воркер описан через main-page feed, без
  search URL.
- `roadmap.md` — Phase 2 переименована в «read-only main-feed
  discovery», добавлена задача наблюдать, как лента реагирует на два
  резюме.
- `open-questions.md` — снят вопрос про стартовые search-фильтры,
  заведён новый: видно ли в одной ленте подбор под оба резюме, или
  только под «основное» / последнее. Решаем эмпирически в Phase 1,
  не теоретизируем заранее.

## Уточнение по двум резюме

Владелец подтвердил эмпирически: его активные резюме —
«LLM-инженер» и «senior fullstack C#/React» — оба учитываются в
одной ленте, попадаются вакансии, релевантные и тому, и другому.

Зафиксировано как [D-009](decisions.md#d-009--2026-05-26--one-shared-feed-llm-routes-each-job-to-a-resume).

резюме, лучшее подходящее (если превышает порог) попадает в отклик.
Соответствующий open question снят.

## Review проекта-донора tech-pulse

Владелец указал на свой актуальный Python-проект
[tech-pulse](https://github.com/vmedvedevpro/tech-pulse), чтобы
позаимствовать стек и стиль. Склонировал в
`c:\Users\vmedv\Documents\git\tech-pulse`, прошёл Explore-агентом.
Ключевое:

- **Стек:** Python 3.11+, `uv`, `pydantic-settings` из `.env`,
  SQLAlchemy 2.0 async + asyncpg, Alembic с async `env.py` и
  ручными именами миграций (`0_initial.py`, `1_…`),
  `loguru` со `.bind()` для контекста, `anthropic.AsyncAnthropic`
  с prompt caching и retry на 529, pytest + pytest-asyncio.
- **Архитектура:** `src/<pkg>/` layout, разделение на
  `domain/persistence/integrations/workers`, репозитории как
  Protocol-интерфейсы, DI собирается в `bootstrap.py`. Тесты —
  fake-объекты в `tests/persistence/fakes.py`, без реальной БД.
- **Docker:** `python:3.12-slim` multi-stage с `uv`, миграции
  накатываются в entrypoint. У нас Dockerfile не нужен (local-only),
  только docker-compose под Postgres.
- **Gap проекта-донора:** нет ruff/mypy/pre-commit. У нас добавим
  как явный improvement (D-011).

Что adoptим: всё перечисленное выше как есть.
Что не adoptим: pgvector, voyageai, python-telegram-bot, yt-dlp,
PyGithub — не нужны для задачи.
Что добавляем уникальное: Playwright + stealth-патчи (тех чего
у tech-pulse нет), `matching/` модуль под три LLM-задачи.

Зафиксировано тремя решениями:

- [D-010](decisions.md#d-010--2026-05-26--adopt-the-tech-pulse-stack-and-layout) — adopt stack.
- [D-011](decisions.md#d-011--2026-05-26--improvements-over-tech-pulse-baseline) — ruff/mypy/pre-commit/gitleaks как improvement.
- [D-012](decisions.md#d-012--2026-05-26--single-env-for-config-no-configtoml) — один `.env`, без `config.toml` (отменяет ранние черновики).

Соответствующие правки внесены в `architecture.md` (§Stack, §Source
layout, §Config & secrets), `conventions.md` (§Code, §Configuration,
§Database, §Testing), `roadmap.md` (Phase 0 расписана пошагово под
этот стек).

## Phase 0 выполнена

Владелец подтвердил решение по browser profile: `./profile/` внутри
репо, gitignored. Зафиксировано как
[D-013](decisions.md#d-013--2026-05-26--browser-profile-lives-at-profile-inside-the-repo-gitignored).

После этого закрыт весь Phase 0 (см. отмеченный чек-лист в
[roadmap.md](../roadmap.md#phase-0--repo-skeleton---done-2026-05-26)).
Создано:

- `pyproject.toml` с зависимостями и конфигами ruff/mypy/pytest,
  console scripts `hhack-feed` / `hhack-chat`.
- `uv.lock` после `uv sync` (uv 0.11.5, Python 3.14 в окружении,
  целевой `requires-python = ">=3.11"`).
- `docker-compose.yml` под `postgres:16` (без pgvector), healthcheck.
- `alembic.ini` + `migrations/env.py` (async вариант, скопирован
  с tech-pulse и переключён на `hhack.config` / `hhack.domain.base`)
  + `script.py.mako` + пустой `versions/`.
- `.env.example` со всеми ключами из
  [architecture.md §5](../architecture.md#5-config--secrets).
- `src/hhack/` скелет: `config.py` (Settings), `logging.py`
  (loguru), `bootstrap.py` (stub), пустые `domain/`, `persistence/`,
  `integrations/`, `matching/`, `workers/`. `persistence/database.py`
  и `domain/base.py` уже на месте.
- `workers/feed.py` и `workers/chat.py` — `async def _run()` + `main()`,
  логируют `started (Phase 0 stub)` и выходят.
- `.gitignore` дополнен (`profile/`, `resumes/*` кроме примеров,
  `*.code-workspace`).
- `.pre-commit-config.yaml` — ruff (fix + format), mypy, gitleaks.
- `README.md` переписан в форме tech-pulse (Requirements /
  Installation / Configuration / Running / Tests), с таблицей env.
- `resumes/example_a.md` и `resumes/example_b.md` как шаблоны.

**Smoke test:** скопировал `.env.example` в `.env` (только для
теста, удалён сразу), запустил `uv run hhack-feed` и
`uv run hhack-chat`, оба напечатали ожидаемый loguru-лог и завершились.

Что осталось пользователю руками:

- `cp .env.example .env` и заполнить `ANTHROPIC_API_KEY`, пути к
  резюме.
- `docker compose up -d` для Postgres.
- `uv run alembic upgrade head` (на момент Phase 0 миграций нет,
  команда отработает no-op).
- `uv run playwright install chromium` (одноразово).
- `uv run pre-commit install` (одноразово, для разработчиков).
